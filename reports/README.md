## Report standard

Each findings report should, where possible, include:

- current state
- evidence level
- claim boundary
- observation
- interpretation
- limitations
- next transition or next experiment

Minimum reporting structure:
- Setup: model, layer, prompts or dataset, and relevant parameters
- Results: metrics with paths to JSON artifacts where applicable
- Observation: what happened before interpretation
- Interpretation: hypothesis, mechanism, or prediction
- Threats to validity: plausible artifacts or confounds
- Next steps: one to three concrete follow-up runs

Template: `templates/template_findings.md`

## Evidence policy

Use explicit evidence labels when substantive claims are made:

- `Exploratory`
- `Supported`
- `Replicated`

## Visibility classes

To reduce confusion for external readers, treat the `reports/` layer as three visibility classes:

### 1. Reviewer-facing scientific documents

- current findings summaries
- major dated findings notes
- key protocol documents when they define interpretation boundaries

### 2. Scientific support documents

- plans
- validation standards
- protocol drafts that shape the active work

### 3. Internal or mixed-purpose documents

- templates
- outreach drafts
- weekly operational plans
- startup/program packets

These may remain in `reports/` for continuity, but they should not be mistaken for the main
scientific review surface. External readers should be routed first to:

- `../README.md`
- `../STATUS.md`
- `../findings/README.md`
- `REVIEWER_INDEX.md`
